Near the end of the July 12 session, the user reported the following bug:
> *"wallet is getting connected but unable to purchase it showing - Stellar simulation failed before producing a signable transaction. console logs - Failed to load resource: the server responded with a status of 502 (Bad Gateway) on `/api/nft/v1/public/checkout/intents`"*

### **Why did the simulation fail?**
In [contracts/collection/src/lib.rs](file:///home/n4bi10p/Buildbox/zexvro/services/nft-service/contracts/collection/src/lib.rs#L223-L242), the contract-level `purchase` function is simulated by the Soroban RPC before signing:
```rust
pub fn purchase(e: Env, buyer: Address, token_id: u32) {
    buyer.require_auth();
    ensure_token_available(&e, token_id);
    let config: SaleConfig = e.storage().instance().get(&DataKey::SaleConfig).unwrap();
    let treasury = read_owner(&e);

    // This operation fails during simulation if the buyer has no trustline or insufficient balance
    token::Client::new(&e, &config.payment_token).transfer(&buyer, &treasury, &config.price);
    Base::mint(&e, &buyer, token_id);
    ...
}
```
1. **USDC Balance/Trustline**: For the simulation to succeed on the Stellar Testnet, the buyer's wallet address (`buyerAddress`) **must**:
    *   Have an established trustline for the payment token (USDC on Testnet: `CBIELTK6YBZJU5UP2WWQEUCYKLPU6AUNZ2BQ4WWFEIE3USCIHMXQDAMA`).
    *   Hold an amount of USDC greater than or equal to the primary sale price.
2. **RPC Simulation Rejection**: If the buyer lacks either the trustline or the balance, the RPC node rejects the transaction simulation, throwing a `HostError`.

### **Why does the API return a generic 502 Bad Gateway?**
In [stellarGateway.ts](file:///home/n4bi10p/Buildbox/zexvro/services/nft-service/api/src/stellarGateway.ts#L68-L105), the API maps simulation errors to HTTP responses. If the error is not one of the pre-mapped contract error codes (like `token_already_minted` or `primary_sale_not_configured`), it falls back to a generic `stellar_simulation_failed` error:
```typescript
return new ApiError(
  502,
  'stellar_simulation_failed',
  'Stellar simulation failed before producing a signable transaction.',
)
```
The raw error stack and message are discarded, making it impossible to see in the frontend or console logs that the error was actually a simple "insufficient buyer balance" or "missing USDC trustline".

### 1. The Freighter "Internal Error" (Fixed)
The error `The wallet encountered an internal error` happened because ZEXVRO's frontend was sending a JSON string to Freighter instead of a binary XDR string.

In the ZEXVRO architecture, the buyer does not sign the transaction envelope—instead, they sign Soroban Authorization Entries (while a ZEXVRO sponsor pays the transaction fee). The backend sent the `AssembledTransaction` to the frontend as a JSON blob. However, the frontend (`stellarWallet.ts`) was passing this JSON string directly to `freighterApi.signTransaction()`. Freighter expects base64 XDR, so it choked on the JSON string and crashed.

I have fixed this by installing `@stellar/stellar-sdk` in the frontend and completely rewriting the `signTransaction` function in [stellarWallet.ts](file:///home/n4bi10p/Buildbox/zexvro/frontend/src/services/nft/stellarWallet.ts). It now detects the JSON payload, reconstructs the `AssembledTransaction`, extracts the specific Soroban Authorization Entries, and uses Freighter's native `signAuthEntry()` to securely prompt the user for their signature.

Yes, you can add the trustline directly in Freighter! The payment token configured for ZEXVRO on testnet is the standard Testnet USDC contract:
`CDLZFC3SYJYDZT7K67VZ75HPJVIEUVNIXF47ZG2FB2RMQQVU2HHGCYSC`

**To add it in Freighter:**
1. Open the Freighter extension and make sure your network is set to **Testnet**.
2. Go to **Manage Assets** (usually at the bottom of the home screen).
3. Click **Add Asset** and paste the USDC Contract ID:
   `CDLZFC3SYJYDZT7K67VZ75HPJVIEUVNIXF47ZG2FB2RMQQVU2HHGCYSC`
4. Confirm and add the trustline.
