# ZEXVRO Stellar Knowledge Base

This document contains detailed specifications, patterns, and code examples for developing on the Stellar network and Soroban Smart Contracts platform. It serves as the primary reference for ZEXVRO's CLI and Web agents.

---

## 1. Stellar Core Concepts

### Accounts & Keypairs
Stellar uses Ed25519 public-key cryptography. Public keys (starting with `G`) identify accounts, and secret keys (starting with `S`) authorize transactions.
- **Minimum Balance:** Accounts require a minimum reserve of 1 XLM (plus 0.5 XLM per sub-entry: trustlines, offers, data entries).
- **Sequence Numbers:** Every transaction must specify the account's current sequence number + 1 to prevent replay attacks.

### Trustlines
Stellar prevents spam by requiring accounts to explicitly opt-in to hold any asset other than XLM. This is called a **Trustline**.
- An account cannot receive or hold an asset (e.g., USDC) without first establishing a trustline to the issuing account.

### Transactions & Operations
A transaction is a bundle of up to 100 operations. Transactions are signed by the source account and submitted to the Horizon API.
- **Operations:** `Payment`, `CreateAccount`, `ChangeTrust`, `ManageBuyOffer`, `ManageData`, `SetOptions` (for multi-sig).

## 2. Soroban Smart Contracts (WASM/Rust)

Soroban is Stellar's WebAssembly-based smart contract platform. Contracts are written in Rust using the `soroban-sdk`.

### Storage Types
Soroban implements state storage separation with distinct pricing and expiration (TTL) mechanics:

1. **Instance Storage:**
   - Shared contract-specific metadata (e.g., admin address, fee settings).
   - Exists as long as the contract instance exists.
   - Shared between all invocations.
2. **Persistent Storage:**
   - Long-lived user-specific data (e.g., user balances, whitelist status).
   - Does not expire automatically but requires proactive TTL management to prevent state archival.
   - Restorable from archival if expired.
3. **Temporary Storage:**
   - Short-lived transaction locks, nonces, or flash loan parameters.
   - Cheap to read and write.
   - Expire and are deleted permanently after a specified ledger threshold. Cannot be restored.

### TTL (Time-To-Live) Management
Developers must manage state expiration. When writing to Persistent or Instance storage, update its TTL:
```rust
// Extend TTL of an instance or entry to prevent archival
env.storage().persistent().extend_ttl(&key, threshold_ledgers, extend_to_ledgers);
```

### Protocol 25/26 ZK Host Functions
Stellar Protocol 25 and 26 introduce native host functions for Zero-Knowledge Proof (ZKP) verification:
- **`bn254_pairing_check`:** Performs pairing checks on the BN254 (alt_bn128) elliptic curve, enabling ultra-efficient verification of Groth16 and Plonk proofs generated by Noir/Circom.
- **`poseidon2_hash`:** Native implementation of the Poseidon2 hash function, designed for SNARK-friendly hashing inside and outside ZK circuits.

## 3. Reference Implementation Patterns

### Soroban Rust Contract Example
```rust
#![no_std]
use soroban_sdk::{contract, contractimpl, Address, Env, Symbol, log};

#[contract]
pub struct ZexvroToken;

#[contractimpl]
impl ZexvroToken {
    pub fn init(env: Env, admin: Address) {
        if env.storage().instance().has(&Symbol::new(&env, "admin")) {
            panic!("already initialized");
        }
        env.storage().instance().set(&Symbol::new(&env, "admin"), &admin);
    }

    pub fn mint(env: Env, to: Address, amount: i128) {
        let admin: Address = env.storage().instance().get(&Symbol::new(&env, "admin")).unwrap();
        admin.require_auth();

        let balance_key = Symbol::new(&env, "bal");
        let mut balance: i128 = env.storage().persistent().get(&to).unwrap_or(0);
        balance += amount;
        
        env.storage().persistent().set(&to, &balance);
        
        // Proactive TTL Extension
        env.storage().persistent().extend_ttl(&to, 50_000, 100_000);
    }
}
```

### Python SDK Example (Horizon)
```python
from stellar_sdk import Server, Keypair, TransactionBuilder, Network

server = Server("https://horizon-testnet.stellar.org")
source_keypair = Keypair.from_secret("S...")
destination = "G..."

account = server.load_account(source_keypair.public_key)
transaction = (
    TransactionBuilder(
        source_account=account,
        network_passphrase=Network.TESTNET_NETWORK_PASSPHRASE,
        base_fee=100,
    )
    .append_payment_op(destination=destination, asset_code="XLM", amount="10.0")
    .set_timeout(30)
    .build()
)

transaction.sign(source_keypair)
response = server.submit_transaction(transaction)
```

### JS/TS SDK Example
```typescript
import { Server, Keypair, TransactionBuilder, Networks, Asset } from 'stellar-sdk';

const server = new Server('https://horizon-testnet.stellar.org');
const sourceKeypair = Keypair.fromSecret('S...');
const destination = 'G...';

const account = await server.loadAccount(sourceKeypair.publicKey());
const transaction = new TransactionBuilder(account, {
  fee: '100',
  networkPassphrase: Networks.TESTNET
})
  .append(TransactionBuilder.payment({
    destination,
    asset: Asset.native(),
    amount: '10.0'
  }))
  .setTimeout(30)
  .build();

transaction.sign(sourceKeypair);
const response = await server.submitTransaction(transaction);
```
